# ポモドーロタイマー 段階的実装計画

## 1. 方針
- 1フェーズを2〜4日で完了できる粒度にする
- 各フェーズで「画面操作 → 状態更新 → 保存/テスト」まで通す
- 1フェーズ1ゴールで進める
- 各フェーズに完了条件（受け入れ条件）を明示する

参照:
- architecture.md
- features.md

## 2. 実装フェーズ
### フェーズ0: 土台整備（半日〜1日）
目的:
- 開発と検証を安定して回せる最小基盤を整える

実装項目:
1. Flaskでトップ画面表示
2. staticとtemplatesの分離
3. Pythonテスト基盤（空テスト1本を含む）
4. Lint/Format方針の確定

完了条件:
1. アプリ起動で画面が表示される
2. テストコマンドが成功する

### フェーズ1: タイマー中核ロジック（1〜2日）
目的:
- UIから独立したタイマーエンジンを完成させる

実装項目:
1. 状態モデル定義（idle, focus, paused など）
2. イベント処理（START, PAUSE, RESUME, RESET）
3. 終了予定時刻ベースの残り時間計算
4. Clock抽象導入（本番時刻とテスト時刻の分離）

完了条件:
1. 状態遷移テストが通る
2. 時間計算の境界値テストが通る

### フェーズ2: MVP UI連携（1〜2日）
目的:
- モック準拠の基本UIで主要操作を成立させる

実装項目:
1. 残り時間表示（mm:ss）
2. 円形プログレスリング表示
3. 開始/一時停止/再開/リセット操作
4. 無効操作の抑止

完了条件:
1. 手動で1サイクル操作が可能
2. 主要DOM更新テストが通る

### フェーズ3: 今日の進捗 + ローカル永続化（1日）
目的:
- MVPとして継続利用できる状態にする

実装項目:
1. 完了ポモドーロ数の集計
2. 集中時間の累積
3. localStorage保存と復元
4. 日付切替時の当日集計リセット

完了条件:
1. 再読み込み後に状態が復元される
2. 完了時のみ統計が増えることをテストで保証

### フェーズ4: Flask API最小導入（1〜2日）
目的:
- フロント単体実装からサーバー連携可能な構造へ移行

実装項目:
1. 統計取得API
2. セッション記録API
3. 設定取得/更新API
4. ルートを薄く保ちサービス層へ委譲

完了条件:
1. API正常系/異常系テストが通る
2. フロントがAPI経由でも動作する

### フェーズ5: 休憩サイクルと設定機能（2日）
目的:
- ポモドーロとしての実用性を高める

実装項目:
1. 短休憩・長休憩遷移
2. 長休憩間隔設定
3. 作業/休憩時間設定UI

完了条件:
1. 連続サイクル（例: 4ポモドーロ）で破綻しない
2. 設定変更が即時反映される

### フェーズ6: 通知・品質仕上げ（1〜2日）
目的:
- 実運用に近い品質へ引き上げる

実装項目:
1. 音通知
2. ブラウザ通知
3. モバイル表示最適化
4. アクセシビリティ最低基準への対応
5. 回帰テスト拡充

完了条件:
1. 通知許可の有無どちらでも動作
2. 主要ユーザーフローの回帰テストが通る

## 3. フェーズ共通ルール
1. 1フェーズ1PRを基本にする
2. 新規機能ごとに最低1本の自動テストを追加
3. 受け入れ確認項目を3〜5件定義する
4. 機能変更時はfeatures.mdを更新する

## 4. 優先順
1. 最優先: フェーズ1〜3（MVP成立）
2. 次点: フェーズ4〜5（拡張性と実用性）
3. 仕上げ: フェーズ6（品質向上）
